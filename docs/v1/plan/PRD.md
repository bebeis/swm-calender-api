# PRD: SWM Teams

## 1. 제품 개요

### 1.1 제품명과 구조

- **루트 제품**: `swm-teams` (SWM Teams)
- **하위 sub-service 2종**
  - **`Calendar`** — 팀 일정 통합 서비스
  - **`Match`** — 팀 서비스 홍보 + 맞베타 매칭 서비스

```
swm-teams (root)
├── Calendar   — 팀 일정 통합 (Google Calendar + When2meet)
└── Match      — 서비스 홍보 + 맞베타 매칭 + 관계 그래프
```

`swm-teams`는 "소마 내부 팀"을 공통 루트 엔티티로 두고, 그 위에 `Calendar`와 `Match` 두 서비스가
독립적으로 얹혀 있는 구조다. 팀은 두 서비스를 각각 선택적으로 사용할 수 있다.

### 1.2 요약

소마(소프트웨어 마에스트로) 내부 팀들을 위한 통합 제품.

- **`Calendar`**: Google Calendar와 When2meet 링크를 서버에서 파싱해 팀 가용성을 하나의 도메인으로 합친다.
- **`Match`**: 팀이 자기 서비스를 홍보하고, 비공개 후보 아이디어를 검토하고, AI 중복 판별로
  출시 서비스/후보 아이디어 간 겹치는 부분을 확인하며, 베타 테스터와 맞베타를 주고받는다.

이 프로젝트는 멘토링 캘린더 싱크 전용이었던 `swm-calender`를 계승해, 팀 단위 협업 전반을 다루는
`swm-teams`로 확장된다. `swm-calender`의 기능은 `Calendar` sub-service로 이전되었고, `Match`는 이번 피봇에서 신규로 추가된다.

### 1.3 배경 및 문제 정의

#### 1.3.1 `Calendar`가 해결할 문제

- 팀원들이 각자 멘토링을 수강하지만, 일정을 팀 Google Calendar에 수동 복제하면 누락이 잦다.
- When2meet처럼 팀 외부 도구로 공유된 가용성 정보가 팀 도메인과 단절되어 있어
  서로 다른 곳을 열어 보며 시간을 맞춰야 한다.

#### 1.3.2 `Match`가 해결할 문제

- **홍보가 흩어진다** — 슬랙/디스코드/노션/단톡에 홍보 글이 분산되어 뒤늦게 찾기 어렵다.
- **아이디어 중복을 늦게 발견한다** — 약 100개 팀의 출시 서비스와 후보 아이디어를 사람이 모두
  비교하기 어렵고, 문제/타겟/해결 방식이 겹치는지 초기에 판단하기 어렵다.
- **후보 아이디어를 공개하기 부담스럽다** — 팀이 아직 확정하지 않은 아이디어는 검증하고 싶지만
  다른 팀에 노출되면 안 된다.
- **맞베타 협의가 수동적이다** — "우리도 해줄게, 너희도 해줘"가 DM으로 흘러 히스토리가 남지 않는다.
- **관계와 기여가 보이지 않는다** — 누가 우리 팀을 테스트했고, 우리가 누구를 도왔는지 한눈에 보이지 않는다.

### 1.4 목표

#### `Calendar`
- 소마 멘토링 일정을 팀 Google Calendar에 원클릭으로 등록한다.
- When2meet 링크를 서버에서 파싱해 팀 가용성을 도메인에 정규화한다.
- 두 소스를 합쳐 하나의 "통합 가용성 뷰"를 제공한다.

#### `Match`
- 팀 단위 **서비스 홍보**를 구조화한다.
- 팀이 **비공개 후보 아이디어**를 입력하고 팀 내부에서 검토할 수 있게 한다.
- AI 기반 **중복 판별**로 공개 출시 서비스와 비공개 후보 아이디어 전체를 비교해 겹치는 지점을 알려준다.
- **베타 요청 / 맞베타 요청**의 전 과정을 기록 가능한 상태로 만든다.
- 완료된 테스트와 **피드백을 자산**으로 누적한다.
- 팀 간 **관계를 그래프**로 가시화한다.

### 1.5 비목표 (Out of Scope)

- 외부(소마 외) 일반 사용자 대상 베타 모집 플랫폼
- 실시간 채팅 / DM 시스템
- 정식 이슈 트래커(Jira / Linear) 대체
- 복잡한 추천 알고리즘 / AI 기반 자동 매칭 고도화
- 네이티브 모바일 앱

### 1.6 핵심 가치

| Sub-service | 가치 | 의미 |
|-------------|------|------|
| `Calendar` | 통합 | Google Calendar와 When2meet 가용성을 하나의 도메인으로 합쳐 본다. |
| `Match` | 발견 | 누가 어떤 서비스를 만들고 있는지 구조적으로 볼 수 있다. |
| `Match` | 검증 | 후보 아이디어가 기존 출시 서비스나 다른 후보 아이디어와 어디서 겹치는지 확인한다. |
| `Match` | 교환 | 단순 모집이 아니라 **맞베타**를 명시적으로 제안/수락할 수 있다. |
| `Match` | 가시화 | 누가 우리를 테스트했고, 우리가 누구를 도왔는지 그래프로 보인다. |

## 2. 타겟 사용자

### 2.1 페르소나

| 페르소나 | 목표 |
|----------|------|
| 팀 OWNER | 팀/서비스/캠페인/캘린더 관리, 들어온 요청 수락·거절 |
| 팀 MEMBER | 다른 팀 탐색, 요청 송신, 테스트 수행, 피드백 작성, 팀 일정 조회 |
| 커뮤니티 전체 | 어떤 팀이 활발히 베타를 교환하는지 그래프로 파악 |

### 2.2 사용자 규모

- 소마 기수당 약 300명 / 약 100팀
- 초기 타겟: 동일 기수 전체
- 확장 타겟: 이전 기수 졸업 팀까지

### 2.3 전제 / 도메인 규칙

1. 서비스는 **소마 내부 사용자 전용**이다.
2. 한 팀은 **하나의 서비스**만 가진다 (`Match` 범위).
   - 단, 기획 피봇팅이 있을 수 있으므로 서비스 프로필은 버전/교체가 가능해야 한다.
3. 한 사용자는 **하나의 팀**에만 속한다.
4. 팀 구성은 특수한 경우(탈주 등)에 한정해 변경될 수 있다.
5. 팀은 `Calendar`와 `Match` 중 한쪽만, 또는 둘 다를 사용할 수 있다 (sub-service는 독립적으로 활성화).

## 3. 핵심 사용자 시나리오

### 3.1 공통 온보딩
1. 팀원이 로그인하고 팀을 만들거나 기존 팀에 합류한다.
2. 팀 프로필 기본값이 세팅되고, `Calendar`/`Match` 각 sub-service를 선택적으로 활성화한다.

### 3.2 `Calendar` 시나리오
1. OWNER가 `Calendar` 활성화 → 서버가 팀 Google Calendar를 자동 생성한다.
2. 팀이 When2meet 링크를 등록하면 서버가 페이지를 파싱해 가용성 슬롯을 도메인에 저장한다.
3. 소마 멘토링 일정은 익스텐션을 통해 팀 캘린더에 원클릭 등록된다.
4. 통합 가용성 뷰에서 팀 공통 가능 시간대를 확인한다.

### 3.3 `Match` 시나리오
1. 팀이 자기 **서비스 소개 페이지**를 만든다 (서비스명, 한 줄 소개, 상세, 플랫폼, 스크린샷, 데모 링크).
2. 확정 전 아이디어는 **비공개 후보 아이디어**로 입력하고 팀 내부에서만 조회한다.
3. 팀은 AI 중복 판별을 실행해 약 100개 팀의 공개 출시 서비스 정보와 비공개 후보 아이디어를 함께 훑고,
   문제/타겟 사용자/해결 방식/핵심 기능 중 어떤 부분이 겹치는지 확인한다.
4. MVP가 나오면 **베타 캠페인**을 연다 (모집 팀 수, 마감, 맞베타 여부 등).
5. 다른 팀이 탐색 화면에서 캠페인을 보고 **단방향 요청** 또는 **맞베타 요청**을 보낸다.
6. 대상 팀이 요청을 수락하면 **테스트 할당**이 생성된다.
7. 테스트 종료 후 **구조화된 피드백**을 남기고, **관계 그래프**가 업데이트된다.

## 4. 유저 플로우

### Flow A [공통]: 온보딩 — 팀 OWNER
1. 로그인 (OAuth)
2. 팀 생성 → 팀 프로필 기본값 세팅
3. (선택) `Calendar` 활성화 → 팀 Google Calendar 자동 생성, When2meet 링크 등록
4. (선택) `Match` 활성화 → 서비스 프로필 작성
5. 초대 코드 발급

### Flow B [공통]: 온보딩 — 팀 MEMBER
1. 로그인
2. 초대 코드 입력 → 팀 합류
3. 권한: 팀 프로필 편집, 활성화된 sub-service의 데이터 조회/편집

### Flow C [Calendar]: Google Calendar Push
1. 소마 멘토링 페이지에서 익스텐션이 멘토링 목록을 파싱
2. 서버가 `events.insert`로 팀 캘린더에 이벤트 생성
3. 팀원 전원의 Google Calendar에 반영

### Flow D [Calendar]: When2meet 통합
1. 팀 프로필에 When2meet 링크 등록
2. 서버가 페이지를 파싱해 가용성 슬롯을 도메인(`Availability`)에 정규화
3. Google `freebusy`와 병합하여 통합 가용성 뷰에 표시

### Flow E [Match]: 서비스 프로필 공개
1. OWNER가 서비스 프로필 작성 (이름, 한 줄 소개, 상세, 플랫폼, 스크린샷, 데모 링크)
2. 공개 시 `Match` 탐색 화면에 노출

### Flow F [Match]: 후보 아이디어 입력 및 중복 판별
1. OWNER가 후보 아이디어를 입력한다 (제목, 요약, 해결하려는 문제, 타겟 사용자, 해결 방식, 플랫폼).
2. 후보 아이디어는 기본값과 유일한 공개 범위가 **비공개**이며, 같은 팀 구성원만 조회할 수 있다.
3. OWNER가 중복 판별을 실행하면 시스템이 공개 출시 서비스 프로필과 모든 후보 아이디어를 비교한다.
4. 결과는 겹치는 부분과 유사도 수준을 보여주되, 다른 팀의 비공개 후보 아이디어 원문·팀명·식별자는 노출하지 않는다.

### Flow G [Match]: 베타 캠페인 생성
1. OWNER가 캠페인 생성 (모집 팀 수, 마감일, 맞베타 여부, 요구 사항)
2. 오픈 → 탐색/검색/필터에 노출

### Flow H [Match]: 요청 / 맞베타 제안
1. 다른 팀이 캠페인 상세에서 **단방향 요청** 또는 **맞베타 요청** 송신
2. 대상 팀 OWNER가 수락/거절/취소
3. 수락 시 **테스트 할당** 생성

### Flow I [Match]: 테스트 실행 & 피드백
1. 테스터 팀이 테스트 수행
2. 완료 시 **구조화된 피드백** 작성 (항목별 점수 + 자유 서술)
3. 관계 그래프에 edge 추가/업데이트

## 5. MVP 범위

### 5.1 `Calendar` In Scope

- 팀 Google Calendar 자동 생성
- 소마 멘토링 → 팀 캘린더 bulk push (중복 방지)
- When2meet 링크 서버 파싱
- Google `freebusy` + When2meet 파싱 결과 병합한 통합 가용성 뷰

### 5.2 `Match` In Scope

- 서비스 프로필 (교체 가능, 피봇 대응)
- 비공개 후보 아이디어 입력 / 조회
- AI 기반 중복 판별 (공개 출시 서비스 + 비공개 후보 아이디어 전체 비교, 비공개 후보 원문 비노출)
- 베타 캠페인 생성 / 수정 / 오픈 / 종료
- 캠페인 탐색 · 검색 · 필터
- 단방향 요청 / 맞베타 요청
- 요청 수락 / 거절 / 취소
- 테스트 할당 관리
- 구조화된 피드백 작성
- 내 팀 중심 그래프 / 전체 그래프
- 인앱 알림

### 5.3 공통 In Scope

- 로그인 / 팀 생성 · 합류 / 팀 프로필
- sub-service별 활성화·비활성화

### 5.4 Out of Scope (MVP)

- 외부 공개 랜딩/마켓플레이스
- 실시간 채팅/DM
- 결제/유료 기능
- 모바일 앱
- 추천 알고리즘 고도화

| US-13 | Match | MEMBER | 테스트 후 구조화된 피드백을 남기고 싶다. | 항목별 점수 + 서술 / 제출 후 관계 edge 추가 |
| US-14 | Match | 전체 | 우리 팀과 연결된 다른 팀을 그래프로 보고 싶다. | 내 팀 중심 n-hop 뷰 / 전체 커뮤니티 뷰 토글 |

### 6.2 Should Have

| ID | Sub-service | 페르소나 | 스토리 | 인수 조건 |
|----|-------------|---------|-------|---------|
| US-15 | Match | OWNER | 서비스 프로필을 피봇 시 교체/버전 관리하고 싶다. | 기존 프로필 보존 + 현재 active 프로필 지정 |

## 7. 시스템 아키텍처 개요

### 7.1 구성 요소

- **클라이언트**: 웹 프론트 (팀 / `Calendar` 뷰 / `Match` 뷰) + 크롬 익스텐션 (소마 사이트 오버레이)
- **백엔드 서버**: Spring Boot on AWS EC2, REST API
- **데이터베이스**: Amazon RDS (MySQL 8.0)
- **외부 API / 소스**:
  - Google Calendar API v3 (`calendars.insert`, `events.insert`, `freebusy.query`) — `Calendar` 전용
  - When2meet 페이지 파싱 (서버 사이드) — `Calendar` 전용
  - AI 중복 판별 모델/프로바이더 — `Match` 후보 아이디어 검증 전용, 서버 사이드 호출

### 7.2 모듈 구성 (논리)

```
swm-teams
├── core/              공통: 인증, 팀, 사용자, 권한
├── calendar/          Calendar sub-service (Google Calendar + When2meet)
└── match/             Match sub-service (서비스 홍보 + 맞베타 + 그래프)
```

> Kotlin 패키지는 역사적 이유로 현재 `swm.calender` 아래에 있으며, 이후 별도 리팩터로
> `swm.teams.core`, `swm.teams.calendar`, `swm.teams.match` 체계로 정리한다.

### 7.3 데이터 흐름

```
[웹 프론트 / 익스텐션]
        │ REST
        ▼
[Spring Boot / EC2]
   ├── core        → [RDS MySQL]
   ├── calendar    → [Google Calendar API] / [When2meet (HTTP 파싱)]
   └── match       → [RDS MySQL] / [AI 중복 판별]
```

### 7.4 핵심 설계 결정

- **공통 루트는 `Team`**: `Team`이 두 sub-service 모두의 참조 대상이다.
- **sub-service 독립 활성화**: 한 팀이 `Calendar`만, `Match`만, 혹은 둘 다 사용할 수 있다.
- **팀 캘린더 자동 생성**: `Calendar` 활성화 시 `calendars.insert`로 팀 캘린더 발급, Calendar ID를 DB에 저장.
- **When2meet 서버 파싱**: 링크 등록 시 서버가 페이지를 파싱해 우리 도메인(`Availability`)으로 정규화.
  개인정보 저장 최소화.
- **통합 가용성 뷰**: Google `freebusy` + When2meet 파싱을 동일 스키마로 합친다.
- **OAuth 토큰 서버 관리**: Client Secret·Refresh Token은 서버에서만 보관.
- **팀 스코프 접근 제어**: 모든 API는 `team_id` + `user_id` 교차 검증.
- **서비스 프로필 교체 가능성**: `Match`에서 한 팀이 하나의 active 서비스 프로필을 갖되, 과거 버전 보존.
- **후보 아이디어 비공개**: 후보 아이디어는 공개 전 검토 데이터로 취급하며, 팀 외부 조회와 탐색 노출을 금지.
- **AI 중복 판별의 비공개 후보 보호**: 중복 판별은 비공개 후보 아이디어를 비교 대상으로 사용하지만,
  다른 팀의 후보 아이디어 원문·팀명·식별자는 결과에 노출하지 않는다.

## 8. 보안 요구사항

### 8.1 인증 및 인가

- Google OAuth 2.0 Authorization Code Flow
- Client Secret은 서버 사이드에서만 보관
- Refresh Token은 AES-256 암호화 후 RDS 저장
- API 요청마다 JWT 토큰 검증
- OAuth scope 최소화: `openid`, `email`, `profile`, `calendar.events`, `calendar.readonly`
  (`calendar.*` scope는 `Calendar` 활성화 팀에서만 요청)

### 8.2 API 보안

- 모든 엔드포인트 인증 필수 (미인증 요청은 401)
- Rate limiting (미인증: 5 req/s, 인증: 20 req/s)
- CORS: `swmaestro.ai` 및 지정된 프론트·익스텐션 origin만 허용
- 외부 URL(When2meet 등) 파싱은 서버에서만 수행 (SSRF 방지 — 허용된 도메인 화이트리스트)
- AI 중복 판별 프로바이더 credential은 서버에서만 관리하고, 클라이언트에 노출하지 않음

### 8.3 데이터 격리

- 팀 간 데이터 격리: 다른 팀의 캠페인/요청/피드백은 공개 설정에 한해서만 조회 가능
- 후보 아이디어는 팀 내부 비공개 데이터이며, 중복 판별 결과에서도 다른 팀 후보 아이디어의 원문과 식별자를 노출하지 않음
- AI 중복 판별 요청/응답 원문은 필요한 최소 범위로만 저장하고, 저장이 필요 없으면 요약 결과만 보존
- 암호화 키는 AWS Secrets Manager에서 관리
- When2meet 파싱 결과는 팀 스코프 내에서만 접근 가능

## 10. 용어

| 용어 | 정의 |
|------|------|
| 팀 (Team) | 소마 내부 3인 단위 팀. 한 사용자는 한 팀. 두 sub-service의 공통 루트. |
| Sub-service | `swm-teams` 하위의 독립 활성화 가능한 기능 단위 (`Calendar`, `Match`). |
| 서비스 프로필 (Service Profile) | `Match`에서 팀이 만드는 서비스 소개 페이지. 팀당 1개(active). |
| 후보 아이디어 (Candidate Idea) | `Match`에서 팀이 확정 전 비공개로 입력하는 서비스 아이디어. |
| 중복 판별 (Duplicate Analysis) | 공개 출시 서비스와 비공개 후보 아이디어를 비교해 겹치는 문제/타겟/해결 방식 등을 요약하는 AI 보조 분석. |
| 캠페인 (Campaign) | `Match`에서 특정 서비스의 베타 모집 단위. |
| 요청 (Request) | `Match`의 단방향 / 맞베타 베타 참여 요청. |
| 테스트 할당 (Assignment) | `Match`에서 요청 수락 시 생성되는 테스트 수행 단위. |
| 피드백 (Feedback) | `Match`의 테스트 종료 후 구조화 결과물. |
| 관계 그래프 (Relation Graph) | `Match`에서 팀을 node, 피드백/교환 이력을 edge로 하는 시각화. |
| 통합 가용성 (Unified Availability) | `Calendar`에서 Google Calendar freebusy + When2meet 파싱을 합친 팀 시간 뷰. |
