# PRD: SWM 멘토링 캘린더 연동 서비스

## 1. 개요

### 1.1 제품명

SWM Calendar Sync

### 1.2 요약

소프트웨어 마에스트로 연수생들이 멘토링 수강 일정을 팀 Google Calendar에 자동 동기화하고, 팀원 간 공통 가용 시간을 소마 사이트에서 바로 확인할 수 있게 하는 크롬 익스텐션 + 백엔드 서비스.

### 1.3 배경 및 문제 정의

소프트웨어 마에스트로에서는 연수생 3명이 한 팀을 이루며, 각자 자유롭게 멘토링 강의를 수강한다. 현재 팀원 간 일정 공유는 수동으로 이루어지고 있어 다음과 같은 문제가 발생한다.

- 팀원이 어떤 멘토링을 수강하는지 파악하려면 매번 물어봐야 한다.
- 팀 미팅 시간을 잡기 위해 각자 캘린더를 비교하며 시간을 소모한다.
- 멘토링 일정을 Google Calendar에 수동 입력하는 과정에서 누락이 발생한다.

### 1.4 목표

- 소마 사이트의 멘토링 일정을 원클릭으로 팀 Google Calendar에 등록 (Push)
- 팀원 전원의 캘린더를 기반으로 공통 가용 시간을 소마 사이트에 오버레이 (Pull)
- 우리 팀뿐 아니라 다른 팀도 사용 가능한 범용 서비스 제공

### 1.5 비목표 (Out of Scope)

- 소마 사이트 자체의 기능 변경 (크롬 익스텐션을 통한 오버레이만 제공)
- 멘토링 수강 신청 대행 (일정 정보 파싱 및 캘린더 동기화만 수행)

## 2. 타겟 사용자

### 2.1 페르소나

**페르소나 A — 팀장 (Team Leader)**

- 소마 연수생, 팀 내 일정 관리를 주도
- 팀 생성, 캘린더 초기 설정, 팀원 초대를 담당
- 팀 미팅 시간 조율에 가장 큰 부담을 느낌

**페르소나 B — 팀원 (Team Member)**

- 소마 연수생, 다양한 멘토링을 자유롭게 수강
- 본인 일정을 팀에 공유하고, 팀 공통 가용 시간을 확인하고 싶음
- 별도 설정 없이 간단하게 참여하길 원함

### 2.2 사용자 규모

- 소마 기수당 약 300명 (약 100팀)
- 초기 타겟: 본인 팀 + 같은 기수 내 5~10팀
- 확장 타겟: 기수 전체

## 3. 유저 플로우

### Flow A: 온보딩 — 팀장

1. 크롬 익스텐션 설치
2. Google OAuth 로그인
3. 팀 이름 입력 후 팀 생성
4. (자동) 서버가 Google Calendar API `calendars.insert`로 팀 전용 캘린더 생성
5. 초대 코드 발급 → 팀원에게 공유

### Flow B: 온보딩 — 팀원

1. 크롬 익스텐션 설치
2. Google OAuth 로그인
3. 초대 코드 입력
4. (자동) 서버가 `acl.insert`로 팀 캘린더 writer 권한 부여
5. Google Calendar 사이드바에 팀 캘린더 자동 표시

### Flow C: Push — 멘토링을 캘린더에 등록

1. 소마 멘토링 페이지 방문
2. 익스텐션 content script가 멘토링 목록을 DOM 파싱
3. 접수 내역 페이지에서 "캘린더에 추가" 버튼 주입
4. 클릭 시 서버가 `events.insert`로 팀 캘린더에 이벤트 생성
5. 팀원 전원의 Google Calendar에 반영

### Flow D: Pull — 캘린더에서 가용 시간 확인

1. 소마 멘토링 페이지 방문
2. 익스텐션이 서버에 `GET /availability` 요청
3. 서버가 팀원 3명의 개인 캘린더에 `freebusy.query` 실행
4. busy 구간 합산 → 공통 가용 시간 계산
5. 멘토링 시간대별 "3/3 참석 가능" 등 뱃지를 소마 페이지에 오버레이

## 5. 시스템 아키텍처 개요

### 5.1 구성 요소

- **크롬 익스텐션**: Content script(소마 사이트 DOM 파싱, UI 오버레이) + Popup(팀 관리, 로그인)
- **백엔드 서버**: Spring Boot on AWS EC2, REST API 제공
- **데이터베이스**: Amazon RDS (MySQL)
- **외부 API**: Google Calendar API v3

### 5.2 데이터 흐름

```
[소마 사이트] ←DOM 파싱→ [크롬 익스텐션] ←REST API→ [Spring Boot / EC2]
                                                          ↕
                                              [RDS]    [Google Calendar API]
```

### 5.3 핵심 설계 결정

- **팀 캘린더 자동 생성**: 팀장이 팀을 만들면 서버가 `calendars.insert`로 팀 전용 캘린더를 자동 생성하여 Calendar ID를 DB에 저장. 사용자가 Calendar ID를 직접 다룰 필요
  없음.
- **OAuth 토큰 서버 관리**: Client Secret과 Refresh Token은 서버에서만 보관. 익스텐션에는 세션 토큰만 전달.
- **EC2 + Spring Boot 선택**: OAuth 토큰 관리, 팀 데이터 CRUD 등 상태 관리가 필요하여 서버리스 대신 전통적 서버 채택.

## 6. 보안 요구사항

### 6.1 인증 및 인가

- Google OAuth 2.0 Authorization Code Flow 사용
- Client Secret은 서버 사이드에서만 보관 (익스텐션에 포함 금지)
- Refresh Token은 AES-256으로 암호화하여 RDS에 저장
- API 요청마다 JWT 토큰 검증
- OAuth scope 최소화: `calendar.events`, `calendar.readonly`

### 6.2 API 보안

- 모든 엔드포인트에 인증 필수 (인증 없는 요청은 401 반환)
- Rate limiting 적용 (인증 전: 초당 5요청, 인증 후: 초당 20요청)
- CORS 설정: `swmaestro.ai` 및 익스텐션 origin만 허용
- API Gateway URL 노출은 불가피하므로, "URL을 알아도 인증 없이는 아무것도 못 하는 구조" 설계

### 6.3 데이터 격리

- A팀 멤버가 B팀 일정을 조회할 수 없도록 팀 단위 접근 제어
- API 호출 시 team_id + user_id 교차 검증
- 암호화 키는 AWS Secrets Manager에서 관리
