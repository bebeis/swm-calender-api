# Data Model: SWM Teams MVP

**Feature**: `001-swm-teams-mvp`
**Date**: 2026-04-28

This model describes domain concepts, not a direct table layout. Storage design must map these concepts through
Exposed tables and repository adapters in `storage:db-core`.

## Shared Value Objects

### UserId, TeamId, CampaignId, CandidateIdeaId, DuplicateAnalysisId, RequestId, AssignmentId, FeedbackId

- Type: opaque identifier
- Validation: must refer to an existing aggregate when used in commands
- API representation: string or numeric id, chosen consistently during implementation

### DateTimeRange

- Fields:
  - `startsAt`: ISO-8601 date-time
  - `endsAt`: ISO-8601 date-time
- Validation:
  - `endsAt` must be after `startsAt`
  - Date-times must include timezone or be normalized by the API boundary

## Team Domain

### User

- Fields:
  - `id`
  - `email`
  - `name`
  - `profileImageUrl`
  - `createdAt`
  - `lastLoginAt`
- Validation:
  - `email` must be verified by Google OAuth
  - user must be eligible through the MVP cohort allowlist
- Relationships:
  - may have at most one active `TeamMember`

### Team

- Fields:
  - `id`
  - `name`
  - `description`
  - `inviteCode`
  - `createdAt`
  - `updatedAt`
- Validation:
  - `name` is required and unique within the active cohort if the implementation introduces cohort scope
  - `inviteCode` is unique, non-empty, and generated by the server
- Relationships:
  - owns many `TeamMember`
  - owns one `SubServiceActivation`

### TeamMember

- Fields:
  - `id`
  - `teamId`
  - `userId`
  - `role`: `OWNER` or `MEMBER`
  - `joinedAt`
  - `removedAt`
- Validation:
  - user cannot have more than one active team membership
  - team must have at least one active OWNER
- State transitions:
  - `ACTIVE -> REMOVED`
  - `MEMBER -> OWNER`
  - `OWNER -> MEMBER` only if another active OWNER remains

### SubServiceActivation

- Fields:
  - `teamId`
  - `calendarEnabled`
  - `matchEnabled`
  - `calendarEnabledAt`
  - `matchEnabledAt`
  - `calendarDisabledAt`
  - `matchDisabledAt`
- Validation:
  - Boolean fields are never null
- Behavior:
  - disabling a sub-service hides it from normal workflows but preserves existing data

## Calendar Domain

### TeamCalendar

- Fields:
  - `id`
  - `teamId`
  - `googleCalendarId`
  - `activatedByUserId`
  - `status`: `ACTIVE`, `AUTH_REQUIRED`, `DISABLED`
  - `createdAt`
  - `updatedAt`
- Validation:
  - team must have `calendarEnabled = true` to create or update active binding
  - `googleCalendarId` is required when status is `ACTIVE`
- State transitions:
  - `AUTH_REQUIRED -> ACTIVE`
  - `ACTIVE -> AUTH_REQUIRED`
  - `ACTIVE -> DISABLED`
  - `DISABLED -> AUTH_REQUIRED`

### MentoringSchedule

- Fields:
  - `id`
  - `teamId`
  - `externalSourceId`
  - `title`
  - `range`: `DateTimeRange`
  - `location`
  - `description`
  - `googleEventId`
  - `createdAt`
- Validation:
  - `externalSourceId` must be unique per team
  - `range.endsAt` must be after `range.startsAt`
- Behavior:
  - repeated bulk push with the same `externalSourceId` must not create a duplicate event

### When2meetLink

- Fields:
  - `id`
  - `teamId`
  - `url`
  - `status`: `PENDING`, `PARSED`, `FAILED`
  - `failureReason`
  - `lastParsedAt`
  - `createdAt`
  - `updatedAt`
- Validation:
  - URL must use HTTPS and an allowed When2meet host
  - `failureReason` is required when status is `FAILED`
- State transitions:
  - `PENDING -> PARSED`
  - `PENDING -> FAILED`
  - `FAILED -> PARSED`
  - `PARSED -> FAILED`

### AvailabilitySlot

- Fields:
  - `id`
  - `teamId`
  - `source`: `GOOGLE_CALENDAR` or `WHEN2MEET`
  - `range`: `DateTimeRange`
  - `availableMemberCount`
  - `busyMemberCount`
  - `createdAt`
- Validation:
  - source is required
  - counts are zero or positive
- Relationships:
  - belongs to a `TeamCalendar` or `When2meetLink` source depending on `source`

### UnifiedAvailability

- Type: read model
- Fields:
  - `teamId`
  - `range`
  - `slots`
  - `generatedAt`
- Behavior:
  - combines Google Calendar busy windows and When2meet availability slots for a query period
  - returns an empty `slots` array when no availability data exists

## Match Domain

### ServiceProfile

- Fields:
  - `id`
  - `teamId`
  - `version`
  - `isActive`
  - `isPublic`
  - `name`
  - `summary`
  - `description`
  - `category`
  - `platforms`
  - `screenshotUrls`
  - `demoUrl`
  - `createdAt`
  - `updatedAt`
- Validation:
  - one active service profile per team
  - `name` and `summary` are required
  - `platforms` is not empty for public profiles
  - `demoUrl` must be a valid URL when present
- State transitions:
  - `draft/non-public -> public`
  - `active -> archived` when a new profile becomes active

### CandidateIdea

- Fields:
  - `id`
  - `teamId`
  - `title`
  - `summary`
  - `problem`
  - `targetUsers`
  - `solution`
  - `category`
  - `platforms`
  - `visibility`: `PRIVATE`
  - `createdByUserId`
  - `createdAt`
  - `updatedAt`
- Validation:
  - team must have `matchEnabled = true`
  - `title`, `summary`, `problem`, `targetUsers`, and `solution` are required
  - `platforms` is not empty
  - `visibility` is always `PRIVATE` in the MVP
- Behavior:
  - visible only to active members of the owning team
  - excluded from public campaign search and service profile discovery
  - available as private corpus input for duplicate analysis with redacted cross-team output

### DuplicateAnalysis

- Fields:
  - `id`
  - `candidateIdeaId`
  - `requestedByTeamId`
  - `requestedByUserId`
  - `status`: `COMPLETED` or `FAILED`
  - `scannedReleasedServiceCount`
  - `scannedCandidateIdeaCount`
  - `failureReason`
  - `generatedAt`
- Validation:
  - requester must be an active member of `requestedByTeamId`
  - `candidateIdeaId` must belong to `requestedByTeamId`
  - scanned counts are zero or positive
- Behavior:
  - compares the candidate idea against active public service profiles/campaign descriptions and private candidate ideas
  - stores summarized result metadata, not raw AI prompts or provider responses
  - failed analyses preserve failure reason without exposing private corpus content

### DuplicateAnalysisMatch

- Fields:
  - `analysisId`
  - `sourceType`: `RELEASED_SERVICE` or `PRIVATE_CANDIDATE_IDEA`
  - `sourceId`
  - `sourceTeamId`
  - `sourceTitle`
  - `sourceDisclosure`: `PUBLIC`, `OWN_TEAM`, or `REDACTED`
  - `similarityLevel`: `LOW`, `MEDIUM`, or `HIGH`
  - `overlapDimensions`
  - `overlapSummary`
- Validation:
  - `overlapDimensions` is not empty
  - released service matches may include public `sourceId`, `sourceTeamId`, and `sourceTitle`
  - own-team candidate idea matches may use `sourceDisclosure = OWN_TEAM`
  - other-team private candidate idea matches must use `sourceDisclosure = REDACTED`
  - redacted matches must not include `sourceId`, `sourceTeamId`, or another team's raw candidate title/content
- Behavior:
  - summarizes overlap by dimensions such as problem, target users, solution, feature, platform, or business model
  - provides decision-support signals only; it does not block campaign creation or profile publishing

### BetaCampaign

- Fields:
  - `id`
  - `teamId`
  - `serviceProfileId`
  - `title`
  - `description`
  - `targetTeamCount`
  - `deadline`
  - `reciprocalAvailable`
  - `requirements`
  - `status`: `DRAFT`, `OPEN`, `CLOSED`
  - `createdAt`
  - `updatedAt`
- Validation:
  - service profile must be active
  - `targetTeamCount` must be positive
  - `deadline` must be in the future when opening
- State transitions:
  - `DRAFT -> OPEN`
  - `OPEN -> CLOSED`
  - `CLOSED -> OPEN` only if deadline remains valid

### MatchRequest

- Fields:
  - `id`
  - `campaignId`
  - `requestingTeamId`
  - `targetTeamId`
  - `type`: `ONE_WAY` or `RECIPROCAL`
  - `status`: `PENDING`, `ACCEPTED`, `REJECTED`, `CANCELED`
  - `message`
  - `createdAt`
  - `updatedAt`
- Validation:
  - requesting team cannot equal target team
  - campaign must be public and open
  - duplicate pending or accepted request from the same requesting team to the same campaign is not allowed
  - reciprocal requests require requesting team to have an active public campaign
- State transitions:
  - `PENDING -> ACCEPTED`
  - `PENDING -> REJECTED`
  - `PENDING -> CANCELED`
  - final states do not transition further

### Assignment

- Fields:
  - `id`
  - `requestId`
  - `testerTeamId`
  - `targetTeamId`
  - `status`: `ASSIGNED`, `FEEDBACK_SUBMITTED`, `CLOSED`
  - `createdAt`
  - `updatedAt`
- Validation:
  - created only from an accepted request
  - tester team and target team must be different
- State transitions:
  - `ASSIGNED -> FEEDBACK_SUBMITTED`
  - `FEEDBACK_SUBMITTED -> CLOSED`

### Feedback

- Fields:
  - `id`
  - `assignmentId`
  - `submittedByTeamId`
  - `submittedByUserId`
  - `scores`
  - `summary`
  - `improvementSuggestion`
  - `submittedAt`
- Validation:
  - each score is an integer from 1 to 5
  - required score keys: `USABILITY`, `VALUE`, `RELIABILITY`, `RECOMMENDATION`
  - `summary` is 10-1000 characters
  - `improvementSuggestion` is at most 1000 characters when present
  - only an assigned tester team member can submit feedback

### Notification

- Fields:
  - `id`
  - `teamId`
  - `type`: `REQUEST_RECEIVED`, `REQUEST_ACCEPTED`, `REQUEST_REJECTED`, `REQUEST_CANCELED`,
    `FEEDBACK_REQUESTED`, `FEEDBACK_SUBMITTED`
  - `referenceType`
  - `referenceId`
  - `message`
  - `readAt`
  - `createdAt`
- Validation:
  - `teamId`, `type`, and reference fields are required
- Behavior:
  - unread notifications have `readAt = null`

### CampaignSearchResult

- Fields:
  - `campaignId`
  - `teamId`
  - `teamName`
  - `serviceName`
  - `serviceSummary`
  - `category`
  - `platforms`
  - `reciprocalAvailable`

### TeamTestHistoryItem

- Fields:
  - `assignmentId`
  - `campaignId`
  - `serviceName`
  - `testerTeamId`
  - `targetTeamId`
  - `assignmentStatus`
  - `feedbackSubmittedAt`
  - `feedbackSummary`

### DuplicateAnalysisResult

- Fields:
  - `analysisId`
  - `candidateIdeaId`
  - `status`
  - `scannedReleasedServiceCount`
  - `scannedCandidateIdeaCount`
  - `matches`
  - `generatedAt`
- Behavior:
  - public released service matches include public source labels
  - private candidate idea matches are redacted and never reveal another team identity or candidate text

## Aggregate Ownership

| Aggregate | Root | Contains |
|-----------|------|----------|
| Team | `Team` | `TeamMember`, `SubServiceActivation` |
| Calendar | `TeamCalendar` | `MentoringSchedule`, `When2meetLink`, `AvailabilitySlot` |
| Match | `ServiceProfile` / `BetaCampaign` / `CandidateIdea` / `DuplicateAnalysis` / `MatchRequest` | `DuplicateAnalysisMatch`, `Assignment`, `Feedback`, `Notification` |

Implementation may split Match repositories by aggregate roots only if the aggregate boundary is explicit and
does not produce duplicate repositories for the same root.
